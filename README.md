

# hi3863-smart-lock

基于 Hi3863 / WS63 平台实现的双终端智能车锁项目，用于参加 2026 年全国大学生嵌入式大赛。

这个项目使用两块 BearPi-Pico H3863 开发板完成双端协同：

- 车载端负责舵机执行、振动检测、蜂鸣报警、锁状态维护以及 WiFi / SLE 状态发送
- 移动端负责 OLED 显示、按键交互、消息提示、本地蜂鸣提示以及 WiFi / SLE 状态接收

项目目标不是单纯做一个“能亮能转”的外设演示，而是做一套能展示双板联动、状态同步、自动开锁和异常报警的智能车锁系统。

## 调试过程中实际遇到的问题

这个项目的调试过程并不顺利，实际遇到的问题很多，下面只记录项目里真实遇到过、并且在代码中实际处理过的问题。

### 1. WiFi 一开始反复连不上

最早移动端 SoftAP 侧的配置使用了不合适的安全参数，导致车载端 STA 侧即使扫描到热点，也很难稳定连上。后来在联调中发现，热点密码和安全类型配置不匹配是关键问题之一，因此后续将移动端热点改为更适合当前调试阶段的配置，并重新梳理了车载端的连接流程。

### 2. 车载端状态包一度发错目标地址

项目中 WiFi 通信不是只要“连上热点”就结束，车载端还需要把状态包真正发到移动端。如果只拿本机获取到的 IP 直接当目标地址，就会出现“看起来连上了，但移动端收不到状态”的情况。联调过程中确实遇到过这种问题，后面重新明确了移动端 AP 地址与 UDP 发包目标，才让 OLED 状态更新链路真正跑通。

### 3. SLE 连上了，但 OLED 还显示 `NO`

项目里一个比较典型的问题是：底层 SLE 连接已经建立，但移动端 OLED 仍然显示未连接。后面排查发现，代码里混用了“底层已连接”和“已准备好收发数据”两个状态，导致显示逻辑和实际连接状态不一致。为解决这个问题，代码中把 `connected` 与 `ready` 两种状态拆开处理，OLED 显示和业务发送不再共用同一个判断条件。

### 4. OLED 时间显示和信号格闪烁时间不对

移动端 OLED 里，收件箱显示时间和 WiFi 信号格闪烁周期在早期版本里偏慢，用户感知明显不对。后面在代码里把相关逻辑从原来的 tick 比较方式，调整为更直接的 `osal_msleep` 步进累计方式，用来保证 `INBOX` 显示时间和闪烁节奏更接近预期。

### 5. 舵机长时间不动作

舵机是项目里调试时间很长的一块。实际遇到过以下几类问题：

- 舵机脉宽范围过激进
- PWM 初始化顺序不对
- `GPIO5` 的 PWM 复用需要严格对上
- 上电时舵机动作时序不合理

后面在调试中逐步确认了 `GPIO5 -> PWM5` 的复用关系，并把舵机脉宽改到更保守的 `1000us ~ 2000us` 范围，同时调整了 PWM 初始化和演示动作顺序。

### 6. 振动传感器逻辑反复打架

振动传感器是本项目里最难调的一部分。调试过程中先后遇到过：

- 触发极性判断不一致
- 下降沿 / 上升沿逻辑混乱
- 线程轮询、中断判定和报警时序互相影响
- 报警触发后系统表现异常

其中一个非常关键的点是：底板上振动传感器所在的 `GPIO13` 外部已经有下拉，传感器导通后应表现为高电平，因此软件判定必须和这一硬件事实一致。后续代码中围绕这个方向进行了多轮收敛和简化，但这部分目前仍然是后续继续优化的重点。

### 7. 振动触发后的报警路径过重

在调试振动报警时，还遇到过一个问题：如果在振动检测路径里直接叠加太多动作，例如复杂打印、立即走通信发送、再切换多个状态，就容易把调试过程变得非常混乱。因此后面在代码上有意识地把“振动触发”和“消息发送”拆开，让报警先发生，再由主循环补发状态与消息，尽量降低链路耦合。

### 8. Windows 下的构建环境问题

除了代码逻辑本身，开发过程中还真实遇到过一些构建环境问题，例如：

- `ninja` 不在当前终端环境中
- `ccache` 相关配置导致构建命令异常
- 工具链进程在某些终端环境里无法正常启动

这些问题不一定来自源码本身，但会直接影响调试效率，因此也属于项目开发过程中的真实困难。
